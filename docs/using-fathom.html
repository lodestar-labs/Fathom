1. Run it locally

git clone https://github.com/lodestar-labs/Fathom.git
cd Fathom
docker compose up

Compose starts SQL Server, waits for it to be healthy, seeds the sample Orders / OrderLines tables, and starts Fathom on port 8080 with the orders sample export registered. Authentication is relaxed in the compose environment so you can explore; production requires a token (section 4).

Three URLs to know:

http://localhost:8080/scalarInteractive API browser — every endpoint, callable from the page.
http://localhost:8080/openapi/v1.jsonThe OpenAPI document, for generating clients.
http://localhost:8080/health/readyReadiness (checks the database). /health/live is process-only.

2. The workflow: define → register → run

  1. Define. Write an export definition — a JSON document naming a root entity (table, key column, fields), optional child entities (with parentKeyColumn), and the filters callers may use. The complete schema is in the export definition reference; the shipped sample is samples/orders/orders.export.json.
  2. Register. Either PUT it to the API (it is validated, persisted to the definition directory, and live immediately), or drop the .json file into the definition directory and restart — both end up in the same place.
  3. Run. GET /api/exports/{name}/run with a format and any filters. The response streams; save it or pipe it onward.
# register (or update) a definition
curl -X PUT http://localhost:8080/api/exports/orders \
     -H "Content-Type: application/json" \
     --data-binary @samples/orders/orders.export.json

# run it
curl -o orders.json "http://localhost:8080/api/exports/orders/run?format=json&country=Denmark&orderDateFrom=2026-01-01"

3. The HTTP API, endpoint by endpoint

All five endpoints live under /api/exports and require authentication. Two error body shapes exist: most errors are { "error": "message" }; a definition that parses but fails structural validation returns { "errors": ["...", "..."] } (one entry per problem, so you can fix them all in one pass).

EndpointWhat it doesResponses
GET /api/exports Lists every registered export: name, version, description, its entity names, and its filters (name, entity, field, operator, required). This is the discovery call — a client that wants to know "what can I ask for, and how can I filter it" reads this. 200 JSON array
GET /api/exports/{name} Returns the full definition document as registered — the exact JSON you'd PUT. 200 the definition · 404 unknown name
PUT /api/exports/{name} Registers or replaces a definition. The body's name must match the route. Validated first, persisted to the definition directory, then made live — a definition that fails validation changes nothing. 200 { name, entities } · 400 { error } (unparseable / name mismatch) or { errors: [] } (structural) · 500 persist failure (registry untouched)
DELETE /api/exports/{name} Unregisters the export and deletes its persisted file. 204 · 404 unknown name
GET /api/exports/{name}/run Runs the export and streams the result. Query parameters: format (json default, xml, csv) plus the definition's filters by name. Filter binding is strict: an unrecognized query key is a 400 naming the supported filters — a typo'd filter can never silently export the whole table. Filter validation and staging happen before the response starts, so bad input is a clean 400, never a broken half-download. The response carries Content-Disposition: attachment with a filename derived from the export name. 200 streamed body · 400 unknown format / unknown query key / invalid filter value · 404 unknown name
OpenAPI note: because filters are defined per export at runtime, they can't appear as typed query parameters in the OpenAPI document for /run. Client generators get the endpoint but not the filter names — clients should read GET /api/exports (or /api/exports/{name}) to discover the available filters for an export.

4. Authentication: getting a token

Fathom authenticates callers with Entra ID (Azure AD) bearer tokens. One-time setup in your tenant:

  1. App registration for Fathom itself (the API): App registrations → New registration. Under Expose an API, set an Application ID URI (e.g. api://fathom) and add a scope (e.g. Export.Run). Put this registration's IDs into Fathom's configuration: AzureAd:TenantId (your tenant GUID — a wildcard like common accepts tokens from any tenant and makes Fathom log a startup warning) and AzureAd:ClientId.
  2. App registration for each caller (your service, a script): give it a client secret or certificate, and grant it access to Fathom's scope (API permissions → Add → your Fathom app).

A service acquiring a token with the client-credentials flow:

# curl version (any language's MSAL library does the same thing)
curl -X POST "https://login.microsoftonline.com/<tenant-id>/oauth2/v2.0/token" \
  -d "client_id=<caller-client-id>" \
  -d "client_secret=<caller-secret>" \
  -d "scope=api://fathom/.default" \
  -d "grant_type=client_credentials"
# → { "access_token": "eyJ..." }

curl -H "Authorization: Bearer eyJ..." \
     "https://your-fathom/api/exports/orders/run?format=csv"
// .NET caller using Microsoft.Identity.Client
var app = ConfidentialClientApplicationBuilder.Create(callerClientId)
    .WithClientSecret(secret)
    .WithAuthority($"https://login.microsoftonline.com/{tenantId}")
    .Build();
var token = await app.AcquireTokenForClient(["api://fathom/.default"]).ExecuteAsync();
http.DefaultRequestHeaders.Authorization = new("Bearer", token.AccessToken);
Authorization is currently coarse: any authenticated caller can both run exports and author definitions. If those are different populations in your organization, put PUT/DELETE /api/exports/* behind a gateway policy or network rule until per-export authorization ships (it's top of the roadmap).

5. Configuration reference

Standard IConfiguration — appsettings.json, environment variables (Fathom__ConnectionString style), Key Vault, all work unchanged.

KeyDefaultMeaning
ConnectionStrings:Fathom (or Fathom:ConnectionString)— The database exports read from. Use a least-privilege login: db_datareader on exactly the tables/views you intend to expose — an authenticated caller can register a definition against any table this login can read. Fathom forces MARS on (MultipleActiveResultSets=true) — required by the level-merging read.
Fathom:DefinitionDirectorydata/exports Where registered definitions persist as {name}.json. Must survive restarts (a volume/persistent share); PUT and file-drop both land here. A file that fails to parse or validate at startup is logged and skipped — it never blocks the host.
Fathom:ExportTimeout00:30:00 Per-SQL-command timeout (TimeSpan format hh:mm:ss): bounds each staging statement and the start of each level's read. It does not bound a client that stalls mid-download (roadmap).
AzureAd:TenantId, AzureAd:ClientId— The Entra ID registration Fathom validates tokens against (section 4). Use your specific tenant GUID.
OTEL_EXPORTER_OTLP_ENDPOINTunset Set it and traces/metrics flow to your collector; nothing else to configure. Span per run (tagged name + outcome), counters for runs and rows per entity, duration histogram — all under source/meter name Fathom.

6. What the output looks like

For the shipped orders sample (an Order root with a Line child):

JSON — ?format=json

Roots stream as one array; each entity's children appear under a property named after the child entity. Declared Date fields render as dates, not midnight timestamps; database NULL is JSON null.

[
  {
    "OrderNumber": "ORD-1001",
    "OrderDate": "2026-03-14",
    "Country": "Denmark",
    "Total": 1249.50,
    "Line": [
      { "LineNumber": 1, "Sku": "WIDGET-A", "Quantity": 3 },
      { "LineNumber": 2, "Sku": "WIDGET-B", "Quantity": 1 }
    ]
  }
]

XML — ?format=xml

The document element is the export's name; each row is an element named after its entity; fields are child elements. Database NULL renders as an empty element marked nil="true".

<orders>
  <Order>
    <OrderNumber>ORD-1001</OrderNumber>
    <OrderDate>2026-03-14</OrderDate>
    <Country>Denmark</Country>
    <Total>1249.50</Total>
    <Line>
      <LineNumber>1</LineNumber>
      <Sku>WIDGET-A</Sku>
      <Quantity>3</Quantity>
    </Line>
  </Order>
</orders>

CSV — ?format=csv

A flat export (no children) is a single text/csv file. A hierarchical export is a zip (application/zip) with one {Entity}.csv per entity. Every row carries a _key column; every non-root row also carries _parentKey, referencing its parent's _key — which is how the hierarchy survives a flat format:

# Order.csv
_key,OrderNumber,OrderDate,Country,Total
1,ORD-1001,2026-03-14,Denmark,1249.50

# Line.csv
_key,_parentKey,LineNumber,Sku,Quantity
1,1,1,WIDGET-A,3
2,1,2,WIDGET-B,1
CSV cells are exported faithfully — a value beginning with =, +, -, or @ is not prefixed or escaped. That keeps the data exact and round-trippable, but a spreadsheet application may interpret such a cell as a formula. If your source data is user-controlled, treat exported CSV as untrusted input in Excel or post-process it.

7. Integrating Fathom into your system

Three patterns cover most uses:

7.1 Your service calls Fathom on demand

Your backend needs data from the operational database in a defined shape — a report, a partner feed, a data handoff. Instead of writing and maintaining that query + serialization code, register a definition once and stream the endpoint:

// any HTTP client; the response streams, so stream it onward — don't buffer
using var response = await http.GetAsync(
    "https://fathom.internal/api/exports/orders/run?format=json&country=Denmark",
    HttpCompletionOption.ResponseHeadersRead);   // ← headers-first: no buffering
response.EnsureSuccessStatusCode();
await using var stream = await response.Content.ReadAsStreamAsync();
// deserialize incrementally, forward to a file/blob, or pipe to your own response

ResponseHeadersRead is the one thing to remember: it makes your client hand you the stream as it arrives instead of buffering the whole export first — which preserves, end to end, the memory behavior Fathom guarantees on its side.

7.2 Scheduled feeds

A nightly partner file is a cron job (or Azure Function timer, or CI schedule) with one curl: acquire a token (section 4), GET /run, drop the output on the destination (SFTP, blob storage, mail). The definition is the contract; changing the feed's shape is a PUT, not a deployment.

7.3 Behind your own API

When external callers shouldn't talk to Fathom directly, proxy it: your endpoint applies its own authorization, then forwards to Fathom and streams the response through. Keep the proxy streaming (no buffering middleware) and pass through Content-Type and Content-Disposition.

Client checklist: discover filters via GET /api/exports rather than hardcoding · treat any 400 body's error text as actionable (it names unknown keys and supported filters) · expect Content-Type to vary by format (application/json, application/xml, text/csv or application/zip) · stream, don't buffer.

8. The Loadstone round-trip

Fathom's hierarchical CSV convention is by design the same one Loadstone's hierarchical CSV import reads: a zip of {Entity}.csv files, rows keyed by _key, children pointing at parents through _parentKey. That makes database-to-database transfer a two-step pipeline with no transformation in between:

# 1. Export a hierarchy from database A
curl -H "Authorization: Bearer $TOKEN_A" -o orders.zip \
  "https://fathom-a.internal/api/exports/orders/run?format=csv&orderDateFrom=2026-01-01"

# 2. Import it into database B (Loadstone: upload against a matching manifest)
curl -H "Authorization: Bearer $TOKEN_B" \
  -F "file=@orders.zip" \
  "https://loadstone-b.internal/api/datasets/orders/files"

The two sides meet in the middle: Fathom's export definition and Loadstone's dataset manifest each describe their own database's shape, and the zip's entity files + key columns are the shared contract. JSON round-trips the same way — Fathom nests children under a property named after the child entity, which is the shape Loadstone's JSON import expects. The fine detail of the import side (manifest authoring, job tracking, dead-lettering) is in Loadstone's validate & import guide.

9. Deployment

Containers

The repo's Dockerfile builds the API image; docker-compose.yml shows the reference wiring (SQL Server health-gated before Fathom starts). For Kubernetes or any orchestrator, the two probes are the integration points:

/health/liveLiveness. Process-only — deliberately independent of the database, so a database blip pulls replicas from rotation (readiness) instead of restart-looping them.
/health/readyReadiness. Checks database reachability (cached a few seconds and single-flighted, so probes can't exhaust the connection pool).

Give Fathom:DefinitionDirectory a persistent volume, or treat definitions as config and PUT them from your pipeline on deploy — both are supported; pick one as the source of truth.

Azure App Service

The README's Running on Azure section is the authoritative sequence: Entra app registration → App Service from the Dockerfile → ConnectionStrings__Fathom (Key Vault reference) + AzureAd__TenantId/ClientId → optional OTLP endpoint for Application Insights. No code changes.

Sizing, honestly

Exports stream end to end — one root subtree in memory at a time regardless of row count. The single exception: hierarchical CSV exports buffer each entity to a temp file before zipping (a zip can only write one entry at a time), so budget bounded temp disk for those. CPU and memory otherwise stay flat as exports grow; the database does the heavy lifting.
