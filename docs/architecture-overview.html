Fathom · Architecture

How Fathom works inside

The pipeline an export request travels — registry, filter resolution, the staged row-numbering trick that makes hierarchies streamable, the N-way merge, and the writers — and the reasoning behind each choice.

The four projects

ProjectRoleDepends on
Fathom.CoreContracts: the export-definition model and its validation, the definition registry and directory store, the dual lookup-provider interfaces, the writer interface.nothing
Fathom.SqlServerThe query engine: SQL generation (staging + reads), filter resolution, the export run lifecycle, the built-in code-list lookup, health check.Core
Fathom.WritersStreaming CSV, JSON, and XML writers over the engine's row stream.Core
Fathom.ApiThe HTTP host: endpoints, Entra ID auth, OpenAPI/Scalar, Serilog, OpenTelemetry, health probes, startup definition loading.all three

The dependency arrows only point inward at Core — the engine and the writers never reference each other; they meet at Core's row-stream and writer contracts. That's what makes a future PostgreSQL provider (roadmap) a new project rather than a rewrite: the writers and the host wouldn't change.

Life of an export request

GET /api/exports/orders/run?format=csv&country=Denmark
        │
        ▼
 1. Registry lookup ──── unknown name → 404
        │
 2. Format resolution ── unknown format → 400 (lists supported)
        │
 3. Strict filter binding ── unknown query key → 400 (lists supported filters)
        │
 4. Filter resolution ── request lookups translate readable values
        │                (Denmark → 208); bad value → 400
        ▼
 5. Staging (per level, root first) ── SELECT … INTO #temp, parameterized
        │
 6. Streamed read ── one reader per level, N-way merge by row number
        ▼
 7. Writer ── CSV / JSON / XML, straight onto the HTTP response

Steps 1–5 all happen before the first response byte. That ordering is a deliberate contract: every way a request can be wrong produces a clean status code and error body; once streaming starts, the request has already proven executable.

The staging trick — why hierarchies stream

The hard problem in hierarchical export is recombination: rows for Order and rows for Line come from different tables, but the output needs each order's lines nested inside it. The classic answers both hurt — a giant JOIN duplicates every parent field onto every child row; loading everything and grouping in memory dies on large exports.

Fathom's answer: stage each level once, with synthetic row numbers, and merge forward-only.

  1. Root staging. The filtered root rows go into a session-scoped temp table, each assigned ROW_NUMBER() OVER (ORDER BY key) — a dense 1..N numbering — plus their real key.
  2. Child staging. Each child level joins its source table to the parent's temp table on the real foreign key, but stores the parent's row number as its correlation, then numbers itself ORDER BY (ParentRowNumber, RealKey). Recursively, level by level.
  3. Merged read. Every level is then read back simultaneously (one reader per level — this is why the connection needs MARS), each ordered by its numbering. A forward-only merge walks the root reader, and for each root row pulls exactly the child rows whose ParentRowNumber matches. No join explosion, no lookahead, no buffering beyond the current root subtree.
Two side benefits fall out. Real keys never leak: correlation runs on synthetic numbers, so primary/foreign key values don't appear in output unless a definition maps them as ordinary fields. Filters compose naturally: a child-level filter simply narrows that level's staging query — parents without surviving children still export (they exist), and children of filtered-out parents never stage (the join can't reach them).

Flat exports (a root with no children) skip staging entirely — one direct SELECT with the same row-numbered shape, because a tempdb round trip would roughly double the I/O for no benefit.

SQL safety, in layers

  • Values: every filter value becomes a SqlParameter — no value is ever concatenated into SQL.
  • Identifiers: schema/table/column names come only from a registered definition, are bracket-quoted with ]-doubling, and are validated at registration (length, control characters). A hostile identifier can't be registered, and even a strange legitimate one can't break quoting.
  • Temp table names: derived from the entity name by sanitizing to [A-Za-z0-9_] plus a deterministic FNV-1a hash of the original name — so two distinct entity names that sanitize identically (A-B vs A.B) can never collide on the same temp table.
  • Least privilege as the outer wall: the engine only ever issues SELECT and SELECT INTO #temp; run it on a login with db_datareader over exactly the exposable tables.

Lookups — the two directions

Reference data usually lives as codes (208) but should export as words (Denmark). Two small interfaces handle both directions of that translation:

InterfaceDirectionUsed when
IExportLookupProviderdatabase value → output stringA field declares "lookup": "countries"; called per (field, row) on the hot path — implementations cache.
IRequestLookupProviderclient value → database valueA filter declares "requestLookup": "countries"; the caller sends ?country=Denmark, the engine binds 208.

The built-in AddCodeListLookup(name, codeType) serves both interfaces from one cached, database-backed instance. Custom providers (an external API, a static dictionary) implement one or both interfaces and register with AddExportLookup<T>() / AddRequestLookup<T>() — a type registered for both directions is shared as a single instance, so there is exactly one cache to warm regardless of direction.

Writers — one row stream, three renderings

The engine produces a single depth-tagged row stream; each writer renders it without ever holding more than the current root subtree:

  • JSON — roots as a streamed array; children grouped under a property named after the child entity; declared dates render as yyyy-MM-dd.
  • XML — document element named after the export; an element per entity row; NULL as nil="true". Names are XML-validated at registration, so a definition that would corrupt the stream can't exist.
  • CSV — flat: one file. Hierarchical: a zip of {Entity}.csv with _key/_parentKey columns (the Loadstone-compatible convention). Because a zip permits only one open entry while rows arrive interleaved, each entity buffers to a temp file first — the one place Fathom uses disk instead of pure streaming; memory stays bounded either way.

Definitions at rest

The registry is the in-memory truth; the definition directory is its durable mirror. A PUT validates → persists → registers, in that order, so a failed persist never leaves a live-but-unsaved definition. At startup the directory reloads: files that fail to parse or validate are logged and skipped — one bad file can't hold the host down. Both authoring styles (API PUT, file drop) converge on the same directory.

Observability & operations

  • One OpenTelemetry span per run, tagged with definition name and outcome (success/cancelled/error); counters for runs and per-entity rows; a duration histogram. Source and meter are both named Fathom.
  • Liveness (/health/live) is process-only by design — the database check lives in readiness (/health/ready, cached + single-flighted), so a database blip drains traffic instead of restart-looping replicas.
  • The run's disposal path tags failure before releasing readers, so the completion metric reflects a mid-stream fault rather than a false success.
